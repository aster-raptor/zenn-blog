# Codebase To Zenn Draft

この Skill は、すでにある原稿の校正ではなく、別リポジトリのコードや設計をもとに Zenn 用の技術記事を書き起こしたいときに使います。

生成物は公開直前の完成稿ではなく、人間がレビューする前提の初稿です。確認済みの事実、解釈、未確認事項は明確に分けて扱ってください。

## 最初に集める情報

記事作成の前に、以下が不足していれば確認します。

- 調査対象のリポジトリまたはディレクトリ
- 何を説明したいかを一文で表した内容
- 想定読者
- その記事を今読む価値
- 長さ、トーン、避けたい話題や見出しの制約

テーマが曖昧なら、先に短いヒアリングを行い、論点を絞ってから下書きに進みます。

## ワークフロー

1. ユーザへの短いヒアリングで、記事の目的、想定読者、新規性を特定する。
2. 対象リポジトリを探索し、記事の目的に関係する実装領域を絞る。
3. 記事として成立しそうな切り口を 2 から 4 件抽出する。
4. 最も強い切り口を 1 つ推奨し、その理由を短く説明する。
5. Zenn 向けの記事パッケージを作る。
   - タイトル案 3 件
   - 想定読者
   - 読後に持ち帰ってほしい要点 3 件
   - 見出し構成
   - 本文の初稿
   - 未解決事項や追記候補
6. 事実と解釈の区別が重要な箇所では、両者を分けて書く。

## コード調査の指針

リポジトリを読むときは、次の観点を優先します。

- どのユーザ課題や業務フローを解決しているか
- 主なデータフローや処理フロー
- どんな設計上のトレードオフがあり、なぜ面白いか
- 一見して分かりにくい実装上の工夫
- 運用上の学び、エッジケース、失敗事例

対象が大きい、または記事の切り口が見えにくい場合は [references/code-reading-checklist.md](references/code-reading-checklist.md) を読むこと。

## ヒアリングの指針

記事テーマがまだ粗い、または読者像や新規性が定まっていない場合は [references/interview-checklist.md](references/interview-checklist.md) を読むこと。

## 下書きの方針

標準スタイルは作業日記ではなく解説型です。「何をしたか」よりも「何が学べるか」が伝わる構成を優先します。

基本の流れは次を推奨します。

1. 問題設定や動機
2. なぜその実装が面白いのか
3. どう動いているのか
4. 重要なコードや設計上の判断
5. 学び、トレードオフ、注意点
6. まとめ

本文を書く前に [references/article-outline-template.md](references/article-outline-template.md) を読むこと。

## Zenn 向けの扱い

- 指示がない限り frontmatter の出力は必須にしない。
- `title` や `topics` の候補を提案するのはよい。
- 見出しごとに読みやすく、実用的な構成を保つ。
- リポジトリの確認範囲が限定的な場合は、断定を避ける。

## 出力ルール

- 新規性を明示する。このコードベースがなぜ記事化に値するのかを説明できる下書きにする。
- ファイル一覧の羅列より、簡潔で意味のある説明を優先する。
- 信頼性が上がる場面では、ファイル名、関数名、設定値、コマンドなどの根拠を添える。
- ユーザの要望がない限り、大きなコード断片は貼らない。
- 最後に、未確認事項、追加検証が必要な点、人間レビューが必要な箇所を短く列挙する。
