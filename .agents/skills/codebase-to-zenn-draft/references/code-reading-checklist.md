# コード調査チェックリスト

コードベースを単なるファイル一覧ではなく、記事の材料に変換するためのチェックリストです。

## まず中核をつかむ

- このリポジトリはどんなユーザ課題や業務フローを支えているか
- 主なエントリポイントはどこか
- 設計意図が強く出ているモジュールはどこか
- 全体像を最短で説明できるデータフローや制御フローは何か

## 記事になる要素を探す

- トレードオフが明確な設計判断
- 想像よりシンプル、または巧妙なワークフロー
- 厳しい制約をどう回避しているか
- 失敗しやすい箇所、エッジケース、運用上の学び
- 他のエンジニアが再利用できるパターン

## 押さえる根拠

- ファイル名やディレクトリ名
- 重要な関数、クラス、コマンド
- 挙動を変える設定値
- 意図が見えるコメントや命名

## 最後に要約する観点

調査後は次を短くまとめる。

- このシステムが何をしているか
- この実装のどこが面白いか
- 読者が何を学べるか
- 焦点を保つために記事から外すべきものは何か
